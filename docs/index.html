04

How key things work

The main request flows, end to end — handy when something breaks and you need to know which service to check.

🔐 Sign in

  1. User clicks Continue with Google — the only self-serve signup (email + password is sign-in only, for accounts created out-of-band).
  2. Google verifies → redirects to /api/auth/callback/google.
  3. NextAuth finds/creates the users row, issues a JWT cookie.
  4. Login is rate-limited per IP (10/min) before any DB/bcrypt work.
  5. Middleware now lets them into /workspace.

📎 Add an item with a file

  1. App asks /api/workspace/upload for a signed token.
  2. Browser uploads the file directly to Vercel Blob.
  3. Blob returns a public URL → saved on the items row in Neon.
  4. File opens later in the in-app viewer (FilePreview).

🔗 Share with a client

  1. Manager creates a share → row in client_shares with a token.
  2. Resend emails the client a magic link.
  3. Client opens /share/:token — no login, read-only.
  4. Middleware allows it via the public-path allowlist.

💳 Upgrade a plan

  1. User picks a plan → Paddle.js opens checkout overlay.
  2. Payment completes on Paddle (cards, tax, receipt email).
  3. Paddle sends a signed webhook → /api/webhooks/paddle.
  4. We verify the signature & write the subscriptions row → plan unlocks.

Rules that aren't obvious (read before changing things):
  • Database changes ship with npm run db:push, not migrations — there are no migration files; db:migrate would try to recreate existing tables.
  • ayuvam.com (apex) is canonical; www must redirect to it, or login/RSC navigation breaks with CORS errors.
  • Paddle Sandbox and Production are separate accounts — separate tokens, price IDs, and webhook secrets. Never mix (e.g. a live_ token with env=sandbox → 403 checkout).
  • NEXT_PUBLIC_* values are compiled into the browser at build time — changing them in Vercel requires a redeploy.
  • Plan limits: workspace count, item count (Free = 50/workspace), and storage are enforced server-side against the workspace owner's plan (companies.createdBy) — so a free-owned workspace caps everyone in it. Member cap is not yet hard-blocked (adding members is admin-only). Entitlements live in lib/billing/plans.ts.
  • Rate limiting (lib/rate-limit.ts) auto-selects its store: Upstash Redis when both UPSTASH_* vars are set, else in-memory. On serverless, in-memory is per-instance (weak) — set Upstash for production. It fails open to memory if Upstash is unreachable, so logins never lock up.
  • AGENTS.md: this Next.js may differ from training data — check node_modules/next/dist/docs/ before writing framework code.
Run it locally: npm install → create .env.local (see §05) → npm run dev → open localhost:3000. Deploy by pushing to main (Vercel builds & ships automatically).
